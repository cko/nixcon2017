Hashsync

Many applications use relational database as a permanent store of mutable structured data. Hashsync is a protocol to synchronise two or more databases.

Hashsync relies on the observation that the final state of any relational database can be reproduced from events as they happened in time.

The challenge is that the final state of the database depends on the order of these events. If we apply the same events on two different databases, possibly on two different computers, in exactly the same order, the two database will be identical.

To record the order of these events, hashsync is using content hashes of the records, current and previous to form a linked list.

The current and previous hash of a record is used to determine the order of events of a given record identified by a code.

Each record - row in a database - has a globally unique code that is used to identify muttable record. It is derived from a random private key using ecdsa curve. This can be used to make sure that only authorised events can be applied in a given database.

In this tutorial, we will use a demo application with just two tables: Person and Address with one Person having multiple Addresses.

In our demo, we will create 4 events from differencess between snapthosts of source database srcdbname.

Then, we will apply these 4 events on an destination database dstdbname.

Database Schema

Most operations that hashsync supports require detailed knowledge of the database schema: table names, field names, field types, relationshipts etc.

Preparing schema segment
cat schema_demo |hsync in dict2json json2pb pb2schemaseg out > schema_demo.seg
cat schema_demo.seg |hsync in pb2adoc out > schema_demo.adoc
cat schema_demo.seg |hsync in schemaseg2local
Storing it into schema_demo.adoc file (content included in this tutorial)
cat schema_demo |hsync in dict2json json2pb pb2schemaseg pb2adoc out > schema_demo.adoc

The exact listing of the schema_demo file used by hsync is available here: Database Schema as python dictionary

To inspect the schema as tables, see Person (hashsync.demo.person)
